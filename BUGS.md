### Expiry Date shows "required" even after selecting a date on Add New Batch
- Reported: 2026-07-10 by Suraj
- Fixed: 2026-07-10
- Root cause: `rgd_inventory_batch_form.xml`'s `expiry_date` field never set an explicit `dateFormat` option, so Magento's `Magento_Ui/js/form/element/date.js` component fell back to its default `outputDateFormat` of `'MM/dd/y'` (e.g. `09/30/2026`) for the value actually submitted to the server. Both `Batch/Save.php`'s `isValidExpiryDate()` and the pre-existing `BatchRepository::validateExpiryDateFormat()` (shipped in Run 1) require a strict `Y-m-d` match (`\DateTime::createFromFormat('Y-m-d', $expiryDate)`), so every real date-picker selection was rejected with "Expiry Date is required and must be a valid date." — a genuine, always-reproducing bug, not user error. Confirmed by reading Magento core's `date.js` directly: `prepareDateTimeFormats()` unconditionally copies `options.dateFormat` into `outputDateFormat` whenever `options.dateFormat` is set (and `initConfig()` always defaults it to `pickerDefaultDateFormat: 'MM/dd/y'` when the field doesn't override it) — so display format and server-submit format cannot be decoupled in this component; only setting `options.dateFormat` itself actually changes what's sent to the server, confirmed against `vendor/magento/module-ui/view/base/web/js/form/element/date.js`. This bug pre-dates today's session (`BatchRepository`'s strict check shipped in Run 1); it likely wasn't caught by Stage 8's QA scenario testing because that test posted an already-ISO-formatted string directly via HTTP rather than driving the real browser date picker.
- Fix: `app/code/vendor/Rgd/Inventory/view/adminhtml/ui_component/rgd_inventory_batch_form.xml` — added `<formElements><date><settings><options><option name="dateFormat" xsi:type="string">y-MM-dd</option></options></settings></date></formElements>` to the `expiry_date` field, forcing the component's `outputDateFormat` to ISO `Y-m-d` (moment `YYYY-MM-DD`) so client and server agree. Trade-off: the date picker now also *displays* dates in `YYYY-MM-DD` form (since this component ties picker-display format and server-submit format together) rather than a locale-style format — a minor, deliberate cosmetic trade-off in favor of correctness, matching what's already stored/exposed elsewhere (DB, GraphQL, grid).
- Verified: XML change validated against `vendor/magento/module-ui/view/base/ui_component/etc/definition/date.xsd` (the `<options><option name="..." xsi:type="...">` shape is schema-correct, same pattern already used for the `source_code` select field in this form). Ran `bin/magento cache:flush` (required after any ui_component XML edit, per the earlier stale-cache bug above). `vendor/bin/phpunit app/code/vendor/Rgd/Inventory/Test/Unit` — 59/59 tests, 182 assertions, all passing (XML-only change, no PHP touched).
- Follow-up (same day, after live testing surfaced the message was still occurring): confirmed via `Magento\Ui\Config\Data` that the compiled config correctly embeds `"options": {"dateFormat": "y-MM-dd"}`, ruling out a stale-cache repeat — so the XML fix alone was landing correctly, but `isValidExpiryDate()`'s byte-exact `Y-m-d` string comparison was still too brittle for real-world browser/date-picker output (confirmed by direct testing: a trailing midnight time component like `2026-09-30 00:00:00`, or a non-zero-padded value like `2026-9-30`, both still fail strict equality even though they're unambiguously valid dates). Hardened the boundary instead of continuing to chase the exact client-side format: added `normalizeExpiryDate()` to `Save.php`, which parses the posted value with PHP's lenient `DateTime` constructor (accepts ISO, US slash format, trailing time components, etc.) and re-emits it as canonical `Y-m-d` before `isValidExpiryDate()` or `BatchRepository`'s own strict check ever see it. Verified against 12 representative inputs (ISO, non-padded, US-slash, ISO-with-time, ISO-8601-with-Z, human-readable "Sep 30, 2026", empty, garbage) — all legitimate dates now normalize and pass; empty/unparseable input still correctly rejected. `vendor/bin/phpunit` 59/59 still passing; phpcs/phpstan clean.
- Operational note: any admin UI date field in this module (or others) needs an explicit `dateFormat` option matching the backend's expected storage format — Magento's date component does not default to ISO. Additionally, server-side date validation on any user-facing form should normalize/parse leniently rather than requiring a byte-exact format match, since client-side date components are inconsistent about exactly what they submit.

### Fatal error saving a batch after adding a new constructor dependency: "Too few arguments... exactly 6 expected"
- Reported: 2026-07-10 by Suraj (live error trace)
- Fixed: 2026-07-10
- Root cause: A prior fix pass added a 6th constructor dependency (`SourceRepositoryInterface`) to `Batch/Save.php` for server-side source_code validation. Magento auto-generates an `Interceptor` class per DI-managed class (`generated/code/Rgd/Inventory/Controller/Adminhtml/Batch/Save/Interceptor.php`) that mirrors the constructor signature verbatim; that generated file was not regenerated after the constructor changed, so it still only accepted 5 arguments while the real class (and the DI container) expected 6 — a `TypeError` at object-instantiation time, before any request code ran.
- Fix: No source change — cleared `generated/code` and `generated/metadata`, then ran `bin/magento setup:di:compile` to regenerate all auto-generated interceptor/factory/proxy classes against the current constructor signatures.
- Verified: `grep` of the regenerated `Interceptor.php` confirms its constructor now accepts and forwards all 6 arguments matching `Save.php`. `bin/magento cache:flush` run afterward. `vendor/bin/phpunit app/code/vendor/Rgd/Inventory/Test/Unit` — 59/59 passing.
- Operational note: any time a DI-managed class's constructor signature changes (new/removed/reordered dependency), run `bin/magento setup:di:compile` (or clear `generated/code`+`generated/metadata` in developer mode) before testing — otherwise the stale generated `Interceptor`/`Proxy`/`Factory` class will fatal-error at instantiation, unrelated to whether the new code itself is correct.

### Batch inventory form is not loading while we click on add new batch (recurrence)
- Reported: 2026-07-10 by Suraj
- Fixed: 2026-07-10
- Root cause: Stale `config`-type cache entry (`Magento\Ui\Config\Data`, cache id `ui_component_configuration_data_rgd_inventory_batch_form`, file `var/cache/mage--9/mage---654_UI_COMPONENT_CONFIGURATION_DATA_RGD_INVENTORY_BATCH_FORM`). This cache held the fully-compiled *pre-refactor* version of `rgd_inventory_batch_form.xml` (imperative `js_config`/`config` arrays), from before the same-day refactor to the declarative `<settings>` form. It was never invalidated after the file was edited — Magento's `config` cache type does not auto-invalidate on ui_component/layout XML changes, in developer mode or otherwise; that requires an explicit `cache:flush`/`cache:clean config`. Confirmed via direct PHP reproduction (bypassing the browser, using the live admin session's own PHP session file to avoid disturbing it) that `Magento\Ui\Config\Data::get('rgd_inventory_batch_form')` returned the OLD `template: templates/form/default` and the OLD imperative `buttons` array, while the fieldset/field data (read via a separate, non-stale path) matched the current file — a mix that only makes sense as a stale cached blob from the old file. Because of this, the page rendered using the OLD `templates/form/default` XHTML skeleton, whose Knockout scope binds to `{{getName()}}.areas` (`rgd_inventory_batch_form.areas`) — a sub-component name that is never registered anywhere in this form's (old or new) component tree. `Magento_Ui/js/core/renderer/layout.js`'s KO `scope` binding waits indefinitely via `uiRegistry.get()` for a registry key that will never resolve, with no error thrown — exactly matching the reported symptom: div renders, no console error, no `mui/index/render` XHR, spinner never clears. (The correct `templates/form/collapsible` template — which the current XML actually specifies — binds scope to `{{getName()}}.{{getName()}}`, i.e. `rgd_inventory_batch_form.rgd_inventory_batch_form`, which *is* a real registered node in the component tree.) The dead `rgd_inventory_batch_form → Magento_Ui/js/form/form` map entry in `requirejs-config.js` was ruled out explicitly — it's an unused, harmless leftover (a requirejs `map` only affects literal `require()` module-id calls, and nothing ever requires the literal string `rgd_inventory_batch_form` as a module).
- Fix: No `app/code` change required — the ui_component XML and PHP were already correct. Ran `bin/magento cache:flush` to clear the stale `config` cache type, which forces `Magento\Ui\Config\Data` to re-read and re-cache the current `rgd_inventory_batch_form.xml` on next request.
- Verified: Direct PHP reproduction of the full production rendering pipeline (`UiComponentFactory::create()` → recursive `prepareComponent()` → `Magento\Ui\Component\Wrapper\UiComponent` block → `toHtml()`), run before and after the cache flush. Before: `data.template` = `templates/form/default`, `data.buttons` = old imperative array, rendered scope = `rgd_inventory_batch_form.areas` (unregistered — matches the bug). After: `data.template` = `templates/form/collapsible`, `data.buttons` = new declarative array (`url`/`class`/`label`/`name` per button), rendered scope = `rgd_inventory_batch_form.rgd_inventory_batch_form` (a real registered node). Also confirmed the cache file regenerates correctly and stays consistent on a subsequent fresh read. `vendor/bin/phpunit app/code/vendor/Rgd/Inventory/Test/Unit` — 49 tests, 161 assertions, all passing (no regressions from the fix, since no code changed).
- Operational note: after editing any `ui_component/*.xml` or layout XML, always run `bin/magento cache:flush` (or `cache:clean config`) before testing in the browser — Magento does not do this automatically, even in developer mode. Worth adding as a reminder to `docs/00-first-time-setup.md` or a Makefile target for contributors who hit this again.

### New Batch button shows spinner forever
- Reported: 2026-07-10 by Suraj
- Fixed: 2026-07-10
- Root cause: Two stacked issues. (1) The listing's "Add New Batch" button referenced a nonexistent PHP class (`Magento\Ui\Component\Buttons\Add`), already corrected in `rgd_inventory_batch_listing.xml` before this log existed. (2) Magento core's `theme.js` calls `$(...).collapse('show')` on the `.entry-edit` wrapper every ui_component form renders, but nothing in this Magento version's requirejs config actually loads the plugin that defines `$.fn.collapse` (Bootstrap 5 also only registers it on `DOMContentLoaded`, too late for `theme.js`) — throwing before the form's own spinner-hiding code could run.
- Fix: `app/code/vendor/Rgd/Inventory/view/adminhtml/requirejs-config.js` (new) — mixin on `js/theme` forcing `jquery/bootstrap/collapse` to load and register `$.fn.collapse` synchronously before `theme.js` executes. `app/code/vendor/Rgd/Inventory/view/adminhtml/web/js/force-collapse-plugin-mixin.js` (new) — the mixin itself. Also `rgd_inventory_batch_form.xml` — added missing `submitUrl` so Save Batch has somewhere to post to.
- Verified: `setup:static-content:deploy -f --area=adminhtml`, then confirmed via anonymous `curl` that `requirejs-config.js`, the mixin file, and `jquery/bootstrap/collapse.js` all serve 200 with the fix present; confirmed in browser console that the `collapse is not a function` error no longer appears.
